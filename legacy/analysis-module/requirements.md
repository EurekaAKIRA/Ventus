# analysis-module 详细需求说明 v3

## 1. 文档目的

本文档用于定义 `analysis-module` 的模块目标、职责边界、输入输出契约、核心子模块、关键中间结构与阶段性实现范围，作为后续毕业设计实现、论文章节撰写和模块迭代的基础设计说明。

当前文档仅聚焦 `analysis-module`，不扩展到整个平台架构，不直接改动现有 `lavague-qa` 执行链。

## 2. 模块名称与定位

### 2.1 模块名称

`analysis-module`

### 2.2 模块定位

`analysis-module` 是自动化测试体系中的前置分析模块，负责将业务需求、产品说明、接口说明、流程文档等输入内容，转换为可审查、可追踪、可供后续执行模块消费的结构化测试场景与 Gherkin 产物。

其核心价值不是“直接执行测试”，而是把原始需求转化为一组质量可控的测试描述，为后续执行链、测试 DSL 或统一测试引擎提供稳定输入。

### 2.3 模块在系统中的角色

- 位于需求输入与测试执行之间
- 负责“理解需求并生成测试描述”
- 不负责“驱动浏览器或接口执行测试”
- 不直接侵入现有 `lavague-qa` 的目录结构和执行逻辑

## 3. 建设目标

### 3.1 总体目标

构建一个低耦合、可迭代、具备轻量文档解析与轻量 RAG 增强能力的需求分析模块，使其能够从文本或文档中提取业务目标与测试点，构造结构化测试场景，并输出标准化 Gherkin `.feature` 文件及相关中间产物。

### 3.2 当前阶段目标

1. 接收自然语言需求或 Markdown/文本说明。
2. 完成文档预处理、结构识别和片段切分。
3. 基于轻量本地索引实现检索增强。
4. 从原始需求与检索上下文中提取测试点。
5. 构建结构化场景对象。
6. 生成可审查的 `.feature` 文件。
7. 输出必要的中间结果、校验报告与任务元数据。

### 3.3 面向毕业设计的目标

- 文档结构清晰，便于论文中描述“模块设计”“核心流程”“数据结构”“关键算法思路”
- 模块边界明确，便于与执行模块进行职责拆分
- 处理流程可追踪，便于展示实验样例、生成过程和结果对比
- 设计留有扩展口，便于后续说明向 DSL、JSON 或更强检索方案演进

## 4. 设计原则

- 低耦合：`analysis-module` 只负责生成分析产物，不直接修改执行链。
- 可审核：所有结果先进入独立目录，默认经过人工检查后再接入执行模块。
- 可追踪：原始输入、解析结果、检索结果、场景结构、Gherkin 文本、校验报告都应保存。
- 渐进式：先实现轻量文档解析和轻量 RAG，不引入重型知识库平台。
- 面向执行：生成内容尽量接近可执行测试语义，而不是只做摘要式文本理解。
- 可降级：当文档信息不完整时，允许输出“最小可执行理解结果”，但需记录不确定项。
- 可扩展：后续可平滑扩展到 JSON/DSL、多来源文档和更强检索方案。

## 5. 模块边界

### 5.1 模块负责

- 接收需求文本或文档内容
- 加载与清洗文本
- 文档结构识别与切分
- 轻量索引构建与检索增强
- 需求理解与测试点提取
- 测试场景构建
- Gherkin `.feature` 生成
- 结果质量校验
- 中间产物与元数据管理

### 5.2 模块当前不负责

- 测试执行
- 浏览器控制与 UI 自动操作
- 接口调度与环境编排
- 权限、租户、用户体系
- 前端交互界面
- 复杂工作流编排平台
- 完整 DSL 平台化建设
- 大规模向量数据库或云端知识库建设

### 5.3 与现有执行模块的关系

- `analysis-module` 与 `lavague-qa` 通过产物文件衔接
- 默认不直接写入 `lavague-qa/features`
- 默认不改动 `lavague-qa` 执行链逻辑
- `lavague-qa` 可视为后续消费方，而非当前模块内部依赖

## 6. 典型处理流程

`analysis-module` 的典型处理链路如下：

1. 接收任务输入与任务元信息。
2. 读取文本或 Markdown 文档。
3. 进行文本清洗、结构识别和段落归一化。
4. 对长文本进行片段切分并建立轻量索引。
5. 根据任务目标或关键主题检索相关片段。
6. 融合原始需求与检索上下文，提取测试点与约束。
7. 将测试点组织为结构化场景对象。
8. 将场景对象转换为 Gherkin `.feature` 文本。
9. 对产物进行结构和语义层面的基础校验。
10. 保存 `.feature`、中间结构、校验报告与任务元数据。

## 7. 输入定义

### 7.1 当前阶段支持的输入类型

1. 单段自然语言需求
2. 多段业务说明文本
3. Markdown 文档
4. 纯文本接口说明
5. 纯文本产品说明
6. 纯文本业务流程说明

### 7.2 后续扩展输入类型

- PDF
- Word
- HTML
- OpenAPI / Swagger
- API schema 文件
- 数据库设计说明

### 7.3 输入最小约束

输入内容至少应满足以下条件之一：

- 明确描述业务目标
- 明确描述用户操作流程
- 明确描述期望结果或系统行为

若输入仅包含极少量关键词、上下文缺失严重或无法识别目标对象，模块应返回“信息不足”的分析结果，并在产物中记录缺失项，而不是伪造完整场景。

### 7.4 输入元数据建议

每个任务建议记录以下元信息：

- `task_name`
- `source_type`
- `source_path`
- `created_at`
- `language`
- `operator` 或触发来源
- `notes`

## 8. 输出定义

### 8.1 主输出

- Gherkin `.feature` 文件

### 8.2 中间输出

- 原始输入副本
- 清洗后的标准化文本
- 文档切分结果
- 检索结果
- 需求解析结果
- 场景结构结果
- 校验报告
- 任务元数据

### 8.3 输出目录建议

```text
analysis-module/
  requirements.md
  generated-features/
  artifacts/
    <task_name>/
      raw/
      parsed/
      retrieval/
      scenarios/
      validation/
      metadata/
```

### 8.4 命名建议

- 目录名优先使用 `task_name + timestamp`
- `.feature` 文件名优先使用语义化英文短名
- 中间结构文件可使用 JSON 命名，如 `parsed_requirement.json`

## 9. 核心中间结构设计

为保证模块可追踪、可调试、可扩展，建议在实现阶段定义统一的数据结构，而不是让各子模块直接传递零散文本。

### 9.1 任务上下文 `TaskContext`

```json
{
  "task_id": "analysis_20260329_001",
  "task_name": "wikipedia_homepage_check",
  "source_type": "md",
  "source_path": "docs/wiki_requirement.md",
  "created_at": "2026-03-29T10:00:00+08:00",
  "language": "zh-CN",
  "status": "parsed"
}
```

### 9.2 文档片段 `DocumentChunk`

```json
{
  "chunk_id": "chunk_001",
  "section_title": "首页展示需求",
  "content": "用户打开首页后应能看到 Wikipedia 标题与搜索框。",
  "source_file": "docs/wiki_requirement.md",
  "source_block": "paragraph_3",
  "tokens_estimate": 32,
  "keywords": ["首页", "标题", "搜索框"]
}
```

### 9.3 需求解析结果 `ParsedRequirement`

```json
{
  "objective": "验证用户打开首页后能看到主标题与关键入口",
  "actors": ["普通用户"],
  "entities": ["首页", "标题", "搜索框"],
  "preconditions": ["用户已进入首页地址"],
  "actions": ["打开首页"],
  "expected_results": ["页面显示 Wikipedia 标题", "页面显示搜索框"],
  "constraints": [],
  "ambiguities": []
}
```

### 9.4 测试场景结构 `ScenarioModel`

```json
{
  "scenario_id": "scenario_001",
  "name": "打开首页后显示主标题与搜索框",
  "goal": "验证首页基础展示",
  "priority": "P1",
  "preconditions": ["用户可访问首页地址"],
  "steps": [
    {"type": "given", "text": "用户进入系统首页"},
    {"type": "when", "text": "页面完成加载"},
    {"type": "then", "text": "页面显示 Wikipedia 主标题"},
    {"type": "and", "text": "页面显示搜索框"}
  ],
  "assertions": ["主标题可见", "搜索框可见"],
  "source_chunks": ["chunk_001"]
}
```

### 9.5 校验报告 `ValidationReport`

```json
{
  "feature_name": "Wikipedia homepage",
  "passed": true,
  "errors": [],
  "warnings": [
    "部分步骤仍缺少更具体的页面定位信息"
  ],
  "metrics": {
    "scenario_count": 1,
    "average_step_count": 4
  }
}
```

## 10. 子模块划分与职责

### 10.1 `input_handler`

职责：

- 接收任务输入
- 规范化任务名和输入来源
- 保存原始输入副本
- 生成任务级目录和上下文对象

输入：

- 用户文本
- 文档路径
- 任务元信息

输出：

- `TaskContext`
- 原始输入副本

### 10.2 `document_loader`

职责：

- 读取 `.txt` 与 `.md`
- 合并多段输入
- 统一字符编码
- 去除明显无效字符

输入：

- 文件路径或原始文本

输出：

- 标准化文本内容

### 10.3 `document_parser`

职责：

- 识别标题、段落、列表、编号结构
- 抽取业务目标、对象、动作、结果
- 标识可能的前置条件、异常条件和约束
- 在信息不足时生成“最小可执行理解结果”

设计说明：

该模块不追求完整语义理解，而是优先抽取对测试场景构建有价值的结构信息。

### 10.4 `chunker`

职责：

- 将长文档切分为适合检索的片段
- 保留章节、段落、来源文件等位置信息
- 尽量避免切分后语义断裂

建议策略：

- 优先按标题和段落切分
- 超长段落再按句子或换行细分
- 单个片段应兼顾完整语义与检索效率

### 10.5 `knowledge_index`

职责：

- 为片段建立轻量本地索引
- 支持任务级隔离
- 为检索模块提供可查询上下文

当前阶段建议：

- 可采用关键词倒排索引、BM25、轻量 embedding 缓存等简单方式
- 不强制引入外部数据库服务

### 10.6 `retriever`

职责：

- 根据任务目标、关键词、场景主题检索相关片段
- 为需求解析和场景构建提供上下文补充
- 输出检索命中结果与分数

约束：

- 检索增强只用于补充理解，不应替代原始需求本身
- 若检索结果与原始需求冲突，应优先保留原始需求并记录冲突点

### 10.7 `requirement_parser`

职责：

- 融合原始需求与检索上下文
- 提取候选测试点
- 识别核心路径、前置条件、断言点、约束项和歧义点
- 将复杂目标拆解为多个单目标场景候选

输出重点：

- `ParsedRequirement`
- 候选测试点列表
- 不确定项列表

### 10.8 `scenario_builder`

职责：

- 将测试点组织为结构化场景
- 控制场景粒度
- 为每个场景补足基本前置条件和验证点

场景拆分原则：

- 一个场景尽量只覆盖一个明确目标
- 核心流程与异常流程应分开建模
- 展示类、跳转类、校验类需求尽量分别成场景

### 10.9 `gherkin_generator`

职责：

- 将 `ScenarioModel` 转换为标准 `.feature` 文本
- 生成 `Feature`、`Scenario`、`Given/When/Then/And`
- 对步骤表达进行统一化和规范化

约束：

- 不生成过于抽象或不可观察的 `Then`
- 不把多个关键动作塞进单个步骤
- 场景标题应体现业务目标而非实现细节

### 10.10 `feature_validator`

职责：

- 校验 Gherkin 结构完整性
- 校验步骤是否过长、过抽象或缺少断言
- 输出错误、警告和基础统计信息

建议校验项：

- 是否包含 `Feature`
- 是否至少包含一个 `Scenario`
- 每个场景是否至少包含一个 `Then`
- 是否存在明显重复步骤
- 是否存在长度过长的场景
- 是否存在无法观察的断言表达

### 10.11 `artifact_manager`

职责：

- 保存所有产物
- 管理任务级目录
- 统一命名规则
- 保证中间文件可追踪和可回溯

## 11. 轻量 RAG 设计说明

### 11.1 设计目标

轻量 RAG 的目标不是构建复杂知识平台，而是在文档较长、需求分散或表达不完整时，帮助模块快速回捞与当前测试目标相关的上下文，提高需求理解与场景生成质量。

### 11.2 当前阶段适用场景

- 需求分散在多个段落中
- 页面说明和预期结果不在同一段中
- 输入文档较长，直接整体喂给模型成本较高
- 需要为场景生成补充页面元素、约束条件或业务术语

### 11.3 当前阶段不追求的能力

- 跨项目知识复用
- 企业级知识库治理
- 多租户共享检索平台
- 海量文档在线向量服务

### 11.4 建议实现路径

1. 先完成可追踪的片段切分。
2. 再实现任务级本地索引。
3. 在生成场景前基于目标词做检索。
4. 将命中片段与原始需求一起送入需求解析阶段。
5. 保存检索命中与分数，便于后续分析效果。

## 12. 需求解析设计说明

### 12.1 解析目标

需求解析的目标是把输入文档转换为测试友好的结构，而不是简单做摘要。其输出应直接服务于场景构建和断言生成。

### 12.2 重点提取内容

- 业务目标
- 参与角色
- 页面或接口对象
- 关键动作
- 状态变化
- 前置条件
- 预期结果
- 约束条件
- 歧义或信息缺口

### 12.3 信息不足时的处理原则

- 可以生成最小场景
- 必须记录缺失信息
- 对无法确定的动作或断言，不应强行编造
- 可在校验报告中标注“需人工补充”

## 13. 场景构建设计说明

### 13.1 场景构建目标

将解析结果转化为粒度合适、逻辑清晰、可被执行模块继续消费的结构化测试场景。

### 13.2 场景最小组成

每个场景至少应包括：

- 场景名称
- 测试目标
- 前置条件
- 操作步骤
- 断言步骤
- 来源依据

### 13.3 场景拆分规则

- 一个需求包含多个用户目标时，拆分为多个场景
- 正常流程与异常流程拆分
- 展示验证与数据操作验证拆分
- 页面进入、关键操作、结果断言尽量形成清晰链路

### 13.4 场景质量要求

- 步骤单一，不混合多个动作
- 断言可观察、可定位、可描述
- 场景名称具备业务语义
- 场景不要依赖隐藏上下文

## 14. Gherkin 生成规范

### 14.1 生成要求

- 生成标准 `Feature`
- 生成一个或多个 `Scenario`
- 使用 `Given / When / Then / And`
- 语句尽量简洁明确
- 步骤应接近执行语义

### 14.2 生成约束

- 不使用“相关页面”“适当按钮”“对应区域”等模糊描述
- 不生成无法观察的断言
- 不将分析结论直接写成技术实现细节
- 不依赖执行模块私有能力才能理解步骤

### 14.3 示例

```gherkin
Feature: Wikipedia homepage display

  Scenario: 用户打开首页后看到主标题与搜索框
    Given 用户进入 Wikipedia 首页
    When 页面加载完成
    Then 页面显示 Wikipedia 主标题
    And 页面显示搜索框
```

## 15. 校验设计说明

校验模块用于保证 `.feature` 文件和场景结构具有最低可用质量。

### 15.1 结构校验

- 是否存在 `Feature`
- 是否存在 `Scenario`
- 是否至少有一个 `Then`
- 步骤顺序是否合理

### 15.2 语义校验

- 断言是否可观察
- 步骤是否过长
- 场景是否过度混合多个目标
- 场景标题与步骤内容是否一致

### 15.3 可追踪校验

- 场景是否保留来源片段
- 是否记录不确定项
- 是否生成校验报告

## 16. 产物管理设计说明

### 16.1 保存范围

- 原始输入
- 清洗文本
- 切分结果
- 检索结果
- 需求解析结果
- 场景结构结果
- `.feature` 文件
- 校验报告
- 任务元数据

### 16.2 管理要求

- 任务级目录隔离
- 文件名避免冲突
- 支持按时间戳追溯
- 支持后续人工审查和实验留档

### 16.3 对毕业设计的价值

产物管理可以为论文中的“实验过程可复现性”“案例分析过程”“结果可追踪性”提供直接支撑。

## 17. 非功能需求

### 17.1 可维护性

- 子模块职责单一
- 处理流程清晰
- 配置与逻辑分离
- 提示词、规则和代码尽量分层管理

### 17.2 可扩展性

后续可扩展：

- PDF/Word 解析
- 更强的检索策略
- DSL/JSON 输出
- 场景评分与排序
- 场景去重与合并
- 与执行模块的自动交接机制

### 17.3 稳定性

- 单一文档解析失败不应导致整个任务崩溃
- LLM 输出异常时应能进行基础清洗

## 24. 当前目录与代码骨架对齐建议

为避免文档设计与实现骨架逐步脱节，建议将当前目录中的最小实现统一对齐到以下约束：

### 24.1 `schemas/` 的作用

- `task_context.schema.json`：约束任务级上下文与任务状态流转。
- `parsed_requirement.schema.json`：约束需求解析结果，明确目标、动作、断言、约束与歧义字段。
- `test_scenario.schema.json`：约束场景集合、检索结果与校验结果的聚合输出。
- `validation_report.schema.json`：约束 `.feature` 校验报告结构。

### 24.2 `src/models.py` 的作用

建议将 `src/models.py` 作为统一中间结构定义层，负责提供：

- `TaskContext`
- `DocumentChunk`
- `RetrievedChunk`
- `ParsedRequirement`
- `ScenarioStep`
- `ScenarioModel`
- `ValidationReport`

这样做的价值是：

- 降低子模块之间直接传递裸字典带来的耦合与歧义。
- 便于后续逐步替换为更严格的校验或更强的实现。
- 便于毕业设计论文中描述“统一中间表示”。

### 24.3 当前阶段推荐的模块输入输出

#### `input_handler`

- 输入：`task_name`、`requirement_text`、`source_type`、`source_path`
- 输出：带 `task_context` 的标准化输入载荷

#### `document_loader`

- 输入：`source_path` 或 `inline_text`
- 输出：统一编码后的原始文本

#### `document_parser`

- 输入：原始文本
- 输出：清洗文本、文档大纲、关键词

#### `chunker`

- 输入：清洗后的文本
- 输出：`DocumentChunk[]`

#### `knowledge_index`

- 输入：`DocumentChunk[]`
- 输出：任务级轻量索引对象

#### `retriever`

- 输入：索引对象、查询词
- 输出：`RetrievedChunk[]`

#### `requirement_parser`

- 输入：原始需求、检索上下文
- 输出：`ParsedRequirement`

#### `scenario_builder`

- 输入：`ParsedRequirement`
- 输出：`ScenarioModel[]`

#### `gherkin_generator`

- 输入：`ScenarioModel[]`
- 输出：`.feature` 文本

#### `feature_validator`

- 输入：`.feature` 文本
- 输出：`ValidationReport`

#### `artifact_manager`

- 输入：各阶段结构与路径参数
- 输出：持久化文件与任务产物目录

## 27. 模块间载荷定义建议

为进一步降低耦合，建议模块间尽量传递“阶段载荷”而不是自由拼装的临时字典。当前阶段至少明确以下载荷边界：

### 27.1 输入归一化载荷

由 `input_handler` 输出：

```json
{
  "task_context": {},
  "raw_requirement": "原始需求文本",
  "source_path": "可选来源路径"
}
```

约束：

- `task_context` 是后续所有阶段的主键入口。
- `raw_requirement` 作为需求解析与问题定位的最终兜底来源。
- 不在该阶段提前注入执行模块相关参数。

### 27.2 文档解析载荷

由 `document_parser` 输出：

```json
{
  "cleaned_text": "清洗后的文本",
  "outline": [],
  "keywords": []
}
```

约束：

- `cleaned_text` 是切分和需求解析的统一文本来源。
- `outline` 保留章节层级与行号，便于后续追踪。
- `keywords` 只作为轻量索引与检索候选，不直接等价于测试点。

### 27.3 检索增强载荷

由 `retriever` 输出：

```json
{
  "query": "当前查询词",
  "hits": [],
  "retrieval_notes": []
}
```

约束：

- 必须保留命中分数和来源片段标识。
- 未命中时也要返回空命中结果，而不是跳过该阶段。
- 如果检索补充与原始需求冲突，应通过 `retrieval_notes` 记录，而不是直接覆盖需求。

### 27.4 场景聚合载荷

由 `scenario_builder` 和 `feature_validator` 共同形成：

```json
{
  "task_context": {},
  "parsed_requirement": {},
  "retrieved_context": [],
  "scenarios": [],
  "validation_report": {}
}
```

约束：

- 该结构应与 `schemas/test_scenario.schema.json` 对齐。
- 它既是当前阶段的主要 JSON 产物，也是后续对接执行模块时更稳定的中间接口。
- `.feature` 文本应被视为展示/执行友好产物，而聚合 JSON 才是更适合调试与追踪的结构产物。

## 28. 场景构建细化建议

在现有“最小场景骨架”基础上，建议继续把场景构建细化为以下规则：

### 28.1 场景来源规则

- 每个场景至少绑定一个来源片段或标记为“仅来自原始需求”。
- 如果一个场景同时依赖多个片段，应保留片段列表而不是只保留最高分片段。
- 来源信息必须可回溯到 `chunk_id`，避免只保留纯文本描述。

### 28.2 场景拆分规则

- 若 `actions` 包含多个并列用户目标，优先拆成多个场景候选。
- 若 `expected_results` 同时覆盖展示、校验、状态变化三类内容，优先拆开。
- 若输入只描述目标未描述动作，可保留单场景，但在歧义项中记录“动作链待补充”。

### 28.3 场景优先级规则

- `P0`：主链路且失败会阻断核心业务。
- `P1`：主链路验证或高频使用场景。
- `P2`：补充性展示或边界校验。
- `P3`：低频异常、可延后处理场景。

当前阶段允许先通过规则启发式给默认值，后续再接入更强评分策略。

## 29. Gherkin 生成细化建议

### 29.1 生成前置整理

在进入 `gherkin_generator` 之前，建议先做一次步骤归一：

- 把前置条件映射到 `Given`
- 把核心动作映射到 `When`
- 把显式断言映射到 `Then`
- 额外断言或补充结果映射到 `And`

这样可以让 `gherkin_generator` 聚焦在格式生成，而不是重新理解语义。

### 29.2 文本生成约束

- 同一场景尽量只保留一个主 `When`
- 不将多个并列断言强行压成单条 `Then`
- 若断言只能写成“系统正确处理”“页面正常显示”之类模糊语句，应在校验阶段给出警告

### 29.3 命名规则建议

- `Feature` 名称优先反映业务主题，而不是文件名
- `Scenario` 名称优先反映用户目标或业务结果
- 步骤文本优先使用中文业务语义，暂不提前绑定执行细节

## 30. 校验与产物管理细化建议

### 30.1 校验分层

建议把校验分成两层：

1. 结构校验：关键字段、步骤顺序、最小组成是否完整。
2. 语义提示：步骤是否过长、断言是否模糊、场景是否过度混合。

当前阶段允许把两类结果统一落到 `ValidationReport`，但在 `errors` 与 `warnings` 中区分。

### 30.2 产物目录细化

建议补充以下命名约定：

- `parsed/cleaned_document.json`
- `parsed/document_outline.json`
- `retrieval/retrieved_chunks.json`
- `scenarios/scenario_bundle.json`
- `validation/validation_report.json`
- `metadata/task_context.json`

好处是：

- 每轮实验输出更容易横向对比。
- 文档与代码更容易围绕固定产物名对齐。
- 后续即使引入更强实现，也不必改变目录组织。

### 30.3 调整建议

如果后续发现 `.feature` 与聚合 JSON 经常不一致，建议把 `ScenarioModel[]` 作为唯一事实来源，`.feature` 始终由其派生，而不是两边分别生成。
