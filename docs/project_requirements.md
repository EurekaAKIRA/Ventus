# 基于 Agent 架构的通用自动化测试平台需求文档

## 1. 文档目标

本文档用于指导当前项目的整体重构工作，依据开题报告将系统拆分为职责清晰、边界明确、可独立演进的多个模块，并为后续的系统设计、编码实现、测试验证和论文撰写提供统一参考。

本文档关注的是整个平台级需求，不再局限于单一 `analysis-module` 或 `lavague-qa` 子项目。

如需查看模块分层、目录重组和 `LaVague` 新定位，请结合 `docs/architecture_replan.md` 一并阅读。

## 2. 项目定位

本项目拟实现一套基于 Agent 架构的通用自动化测试平台原型，核心目标是打通以下闭环：

`需求理解 -> 测试用例生成 -> 测试执行 -> 结果分析 -> 可视化展示`

系统重点支持：

- 基于需求文档、接口文档、业务说明的自动测试点提取
- 基于结构化描述的测试用例生成
- 面向接口功能验证与性能压测的统一执行调度
- 多步骤测试场景中的上下文数据传递
- 测试结果分析与可视化展示

当前阶段不把 UI 自动化作为平台核心主线，而是保留为后续可扩展执行能力。

## 3. 重构目标

### 3.1 总体目标

将当前仓库重构为“平台主线清晰、模块职责单一、输入输出稳定、便于毕业设计实现和展示”的系统结构。

### 3.2 本轮重构目标

本轮先完成以下内容：

- 明确平台模块划分与边界
- 统一术语和主流程定义
- 明确每个模块的职责、输入、输出和依赖关系
- 给出推荐的目录重组方案
- 形成后续编码重构的基线需求文档

本轮暂不要求一次性完成所有代码迁移与实现。

## 4. 系统主流程

系统主流程定义如下：

1. 用户提交需求文档、接口说明或业务描述
2. 需求解析模块对输入内容进行清洗、切分、语义提取和检索增强
3. 测试用例生成模块产出结构化测试场景与统一测试描述
4. 执行调度模块根据测试描述选择本地功能执行或云端压测执行
5. 上下文数据模块在多步骤执行过程中维护共享参数与中间结果
6. 结果分析模块汇总执行结果、统计指标并生成展示数据
7. 前端展示模块呈现任务配置、执行状态、结果明细与统计图表

## 5. 核心模块划分

### 5.1 模块一：统一接入与任务管理模块

#### 模块定位

负责接收测试任务、管理任务基础信息、管理测试项目与运行上下文，是整个平台的统一入口。

#### 核心职责

- 接收用户创建的测试任务
- 管理任务名称、来源、描述、标签和状态
- 维护任务级元数据
- 管理测试项目、环境、目标地址和基础配置
- 为后续模块分配统一任务 ID

#### 输入

- 任务名称
- 需求文档或需求文本
- 目标系统地址或接口信息
- 运行环境配置

#### 输出

- 标准化任务对象
- 任务上下文元数据

#### 当前实现映射

- 可复用 `analysis-module` 中的任务输入与产物管理思路
- 后续需要平台化，不应继续只停留在单模块 CLI 输入

### 5.2 模块二：需求解析与知识检索模块

#### 模块定位

负责将自然语言需求、接口说明、产品文档等输入转换为可用于测试生成的结构化需求表示。

#### 核心职责

- 文档加载与清洗
- 文档结构识别
- 文本切分与片段化
- 轻量检索增强
- 测试目标、角色、动作、约束、断言点提取
- 不确定项记录

#### 输入

- 原始需求文本
- Markdown、txt 等文档内容
- 后续可扩展 Word、PDF、OpenAPI 等输入

#### 输出

- 清洗后的文档内容
- 文档片段集合
- 检索命中结果
- 结构化需求对象

#### 当前实现映射

- 主要对应 `analysis-module`
- 当前已有的 `document_loader`、`document_parser`、`chunker`、`retriever`、`requirement_parser` 可以作为该模块初始骨架

### 5.3 模块三：测试用例生成模块

#### 模块定位

负责根据结构化需求生成统一测试描述，作为执行层的标准输入。

#### 核心职责

- 根据需求提取结果生成测试场景
- 拆分主流程、异常流程和边界流程
- 生成统一 DSL 或 JSON 测试描述
- 按需生成 Gherkin `.feature` 作为展示和兼容产物
- 标注测试优先级、前置条件、断言和依赖关系

#### 输入

- 结构化需求对象
- 检索增强上下文

#### 输出

- 场景集合
- 统一测试描述文件
- Gherkin `.feature`

#### 当前实现映射

- 当前 `analysis-module` 中的 `scenario_builder`、`gherkin_generator` 属于这一层
- 现阶段虽然已有 `.feature` 生成能力，但后续应提升为“统一 DSL/JSON 为主，Gherkin 为辅”

### 5.4 模块四：执行引擎模块

#### 模块定位

负责消费统一测试描述，完成测试任务的真正执行，是平台的核心运行层。

#### 子模块划分

- 本地功能执行子模块
- 云端压测适配子模块

#### 核心职责

- 解析统一测试描述
- 执行接口请求、流程步骤和断言
- 支持串行、并发和异步执行
- 支持本地功能验证
- 支持对接第三方云压测平台 API
- 输出标准化执行结果

#### 输入

- 测试描述 DSL/JSON
- 运行环境配置
- 上下文数据

#### 输出

- 步骤级执行结果
- 场景级执行结果
- 汇总后的任务执行结果

#### 当前实现映射

- `lavague-qa` 更接近“面向 Web/Gherkin 的执行消费层”
- `lavague-tests` 提供了部分测试运行相关能力
- 但当前仓库还没有形成报告中所要求的“本地接口执行 + 云端压测适配”的统一执行引擎，需要重构补齐

### 5.5 模块五：上下文数据管理模块

#### 模块定位

负责解决多步骤测试中的参数传递和状态共享问题，是执行链路中的基础支撑模块。

#### 核心职责

- 存储任务执行过程中的共享数据
- 管理 Token、订单号、用户 ID 等动态参数
- 支持步骤间变量引用
- 支持作用域隔离
- 支持运行期读写和回溯

#### 输入

- 上一步骤执行结果
- 运行过程产生的临时变量

#### 输出

- 可供后续步骤引用的上下文数据视图

#### 技术建议

- MVP 阶段可先采用内存对象或本地缓存
- 第二阶段可引入 Redis 作为上下文总线

### 5.6 模块六：结果分析与质量评估模块

#### 模块定位

负责对执行结果进行标准化清洗、聚合统计和质量评估。

#### 核心职责

- 汇总各执行步骤结果
- 统计成功率、失败率、响应时间和并发指标
- 区分功能性失败与性能瓶颈
- 生成统一分析结果对象
- 产出可视化展示所需数据

#### 输入

- 本地执行结果
- 云端压测结果

#### 输出

- 分析报告
- 指标统计结果
- 图表数据源

### 5.7 模块七：可视化展示与交互模块

#### 模块定位

负责平台前端交互、任务配置和结果展示，是面向用户的界面层。

#### 核心职责

- 提供任务创建和配置界面
- 展示需求解析结果与生成用例
- 展示执行进度和日志
- 展示结果统计图表与明细
- 提供历史任务查询能力

#### 输入

- 任务数据
- 执行状态
- 结果分析数据

#### 输出

- 可交互的前端页面

### 5.8 模块八：用户与权限管理模块

#### 模块定位

负责用户身份、项目隔离和敏感配置管理，为后续多用户扩展提供基础。

#### 核心职责

- 用户登录与身份标识
- 项目级权限控制
- 接口密钥、访问令牌和环境变量配置管理
- 多环境隔离

#### 说明

- 从开题报告看这是平台需要具备的组成部分
- 但在毕业设计 MVP 中可先保留最小实现，例如单用户 + 简单配置管理

## 6. 模块间依赖关系

推荐依赖顺序如下：

`任务管理 -> 需求解析 -> 用例生成 -> 执行引擎 -> 结果分析 -> 可视化展示`

其中：

- 上下文数据管理模块服务于执行引擎
- 用户与权限管理模块服务于任务管理与执行配置
- UI 自动化能力如需保留，应作为执行引擎的一个扩展适配器，而不是平台主流程的中心模块

## 7. 平台统一输入输出约束

### 7.1 上游统一输入

平台需要统一接收以下信息：

- `task_id`
- `task_name`
- `source_type`
- `source_path` 或 `requirement_text`
- `target_system`
- `environment`
- `operator`
- `created_at`

### 7.2 中间统一对象

建议平台后续统一以下核心中间结构：

- `TaskContext`
- `ParsedRequirement`
- `TestScenario`
- `TestCaseDSL`
- `ExecutionResult`
- `ValidationReport`
- `AnalysisReport`

### 7.3 下游统一输出

平台最终应输出：

- 结构化测试描述
- 执行日志
- 执行结果汇总
- 分析报告
- 可视化展示数据

## 9. 当前仓库到目标模块的映射建议

### 9.1 可直接保留并演进的部分

- `analysis-module` -> `requirement_analysis` + `case_generation`
- `lavague-qa` -> `execution_engine/web_ui_adapter` 的候选基础
- `lavague-tests` -> 执行验证与测试基建的候选基础

### 9.2 需要新建或重点补齐的部分

- `task_center`
- `context_bus`
- `result_analysis`
- `auth`
- `frontend`
- `cloud_load_api` 适配层

### 9.3 当前不建议直接继承的部分

- 不建议继续以 `lavague-qa` 作为整个平台核心
- 不建议让 `.feature` 文件成为唯一事实来源
- 不建议把 UI 自动化链路等同于平台主线

## 10. MVP 范围

结合开题报告与毕业设计可控性，建议 MVP 聚焦以下能力：

- 需求文本或 Markdown 输入
- 需求解析与测试点提取
- 场景生成与统一测试描述生成
- 本地接口功能执行
- 基础上下文参数传递
- 基础执行结果汇总
- 简单结果展示页面

MVP 阶段可暂缓：

- 完整多用户权限体系
- 完整云压测平台对接
- 完整 UI 自动化执行
- 复杂文档格式全量支持
- 大规模知识库平台化建设

## 11. 非功能需求

### 11.1 可维护性

- 模块职责单一
- 输入输出稳定
- 配置与逻辑分离
- 支持独立测试

### 11.2 可扩展性

- 支持新增输入格式
- 支持新增执行适配器
- 支持从规则解析升级到 LLM 增强解析
- 支持从本地执行扩展到云端执行

### 11.3 可追踪性

- 原始输入可回溯
- 中间产物可落盘
- 执行日志可定位
- 结果分析过程可复现

### 11.4 稳定性

- 单模块失败不应导致整个平台完全不可用
- 检索增强失败时应允许降级
- 可视化展示不应阻塞执行主链路

## 12. 分阶段实施建议

### 第一阶段：建立平台骨架

- 完成总需求文档
- 明确模块目录和边界
- 统一核心数据模型

### 第二阶段：重构需求解析与用例生成

- 迁移 `analysis-module`
- 抽离 `requirement_analysis` 和 `case_generation`
- 将 `.feature` 降级为派生产物

### 第三阶段：建设执行主链路

- 设计统一 DSL/JSON 执行协议
- 实现本地功能执行器
- 接入上下文数据管理

### 第四阶段：建设结果分析与展示

- 统一执行结果模型
- 生成统计指标
- 完成基础前端展示

### 第五阶段：扩展能力

- 云端压测适配
- UI 自动化适配
- 权限管理增强

## 13. 本轮重构结论

基于开题报告，当前项目后续重构应围绕以下主模块展开：

- 统一接入与任务管理
- 需求解析与知识检索
- 测试用例生成
- 执行引擎
- 上下文数据管理
- 结果分析与质量评估
- 可视化展示与交互
- 用户与权限管理

其中：

- `analysis-module` 不再被视为完整平台，只是“需求解析 + 用例生成”子系统
- `lavague-qa` 不再被视为平台主线，只保留为“执行适配能力候选”
- UI 自动化不是当前重构主线，应作为执行层扩展能力保留接口
