# API 需求文档编写规范 v1

## 适用范围

- 适用于本项目中面向 **API 场景生成、DSL 生成、执行与报告** 的需求文档。
- 主要服务于 **API E2E / 接口链路烟测 / 回归测试** 场景。
- **不覆盖** UI 自动化、探索式测试记录、纯业务 PRD、无接口细节的产品说明文档。

---

## 规范目标

这份规范的目标不是让文档作者手工写测试脚本，而是让文档提供“系统自动生成场景、用例并一键测试”所需的最小完备信息。

符合规范的文档，应能支持系统自动完成以下链路：

1. 识别测试目标和覆盖范围
2. 提取接口清单
3. 识别资源生命周期与依赖链
4. 自动生成候选场景
5. 自动补步骤、上下文与断言
6. 生成 DSL
7. 发起执行并输出报告

因此，文档应更偏 **功能描述、接口能力、资源关系、预期效果**，而不是要求作者手工写完整测试步骤和细粒度断言。

---

## 最小必填结构

每份 API 需求文档至少应包含以下结构：

1. 文档定位
2. 环境信息
3. 鉴权与全局约束
4. 资源依赖说明
5. 接口清单
6. 能力链路
7. 预期效果

推荐骨架如下：

```md
# 文档标题

## 文档定位

## 环境信息

## 鉴权与全局约束

## 资源依赖说明

## 接口清单

### Scenario: 某能力链路

## 预期效果
```

## 硬规则

以下规则属于 v1 的强约束，违反后文档应视为不完整输入。

### 1. 文档必须清楚说明要测什么

- 必须说明本次验证的业务目标或功能范围。
- 必须说明覆盖的是哪些核心接口或能力链路。
- 禁止整篇文档只罗列请求样例，而没有功能目标描述。

### 2. 环境与全局约束必须集中声明

- 必须提供 Base URL。
- 如有统一响应 envelope、默认请求头、特殊状态码规则，必须集中写在“环境信息”或“鉴权与全局约束”中。
- 鉴权方式、全局 Header、Base URL 不应散落在多个功能描述中。

### 3. 接口必须使用稳定标识

- 接口必须显式写出 `` `METHOD /path` ``。
- 不允许只写中文描述替代接口标识，例如“调用创建订单接口”不能替代 `` `POST /orders` ``。
- 接口路径、query 参数、占位符写法应尽量统一。

正确：

```md
- `POST /booking`
- `GET /booking/{id}`
```

错误：

```md
- 创建 booking 接口
- 查询详情接口
```

### 4. 依赖活资源的接口必须声明资源来源

对 `detail / update / patch / delete` 这类依赖活资源的接口，文档必须明确以下之一：

- 同链路 `create` 创建资源
- 同资源 `list / detail` 获取并保存 id
- 文档显式声明预置资源

若三者都没有，文档视为不完整，不建议生成独立场景。

### 5. 能力链路必须说明依赖关系

- 文档必须说明哪些接口是独立能力，哪些接口依赖其他接口或依赖已有资源。
- 若一个能力链路包含鉴权、创建、更新、删除等前后关系，必须用文字说明依赖顺序。
- 不要求作者手工拆每一步测试步骤，但必须写清功能之间的前置关系。

### 6. 预期效果必须是可验证的

- 文档必须说明“达到什么效果才算符合预期”。
- 预期效果可以是功能语义，不要求写成细粒度断言脚本。
- 禁止只写“返回正常”“成功即可”这类不可验证表述。

正确：

```md
- 创建成功后应返回新 booking 标识
- 删除后再次读取应返回资源不存在
```

错误：

```md
- 接口应正常工作
```

## 推荐规则

以下规则不是强制项，但强烈建议遵守，以提高自动生成质量。

### 1. 使用固定章节名

推荐使用以下章节名，便于规则解析和后续质量校验：

- `## 文档定位`
- `## 环境信息`
- `## 鉴权与全局约束`
- `## 资源依赖说明`
- `## 接口清单`
- `### Scenario: ...`
- `## 预期效果`

### 2. 用 Scenario 表达能力链路

推荐把 `Scenario` 写成“能力链路说明”，而不是完整测试脚本。

例如：

```md
### Scenario: Booking 生命周期管理

覆盖 booking 的创建、详情读取、全量更新、部分更新与删除能力。
```

### 3. 场景中推荐补三类信息

推荐在 `Scenario` 下补充：

- `**涉及接口:**`
- `**关键依赖:**`
- `**资源来源:**`

这些字段能帮助系统更稳定地形成依赖链，但不要求作者手工写完整步骤。

### 4. 接口清单描述功能，不描述测试动作

推荐每个接口至少说明：

- 方法
- 路径
- 功能
- 是否鉴权
- 请求体关键字段或请求语义
- 结果语义

### 5. 预期效果保持轻量

推荐写“预期效果”而不是手工拆状态码断言、字段断言、业务值断言。

例如：

```md
- 部分更新后，未修改字段应保持不变
- 过滤查询应返回合法数组结果
```

系统应基于这些预期效果自动补充具体断言。

## 能力链路写法

推荐的能力链路结构如下：

```md
### Scenario: 创建后更新再删除

覆盖 booking 的创建、详情读取、全量更新、部分更新与删除能力。

**涉及接口:** `POST /booking`、`GET /booking/{id}`、`PUT /booking/{id}`、`PATCH /booking/{id}`、`DELETE /booking/{id}`

**关键依赖:** `GET /booking/{id}`、`PUT /booking/{id}`、`PATCH /booking/{id}`、`DELETE /booking/{id}` 依赖已存在 booking

**资源来源:** 默认通过 `POST /booking` 创建 booking
```

注意：

- `Scenario` 主要描述能力链路，而不是要求作者细写每个测试动作。
- 一个 `Scenario` 可以覆盖一个完整资源生命周期，也可以覆盖一组同类只读能力。
- 同一链路下的接口应尽量按资源生命周期顺序描述。

## 接口清单写法

推荐的接口清单结构如下：

```md
### 接口：创建 booking

- 方法：`POST`
- 路径：`/booking`
- 功能：创建 booking
- 鉴权：否
- 请求语义：提交姓名、价格、入住离店日期等信息
- 结果语义：返回新 booking 标识及创建结果
```

```md
### 接口：删除 booking

- 方法：`DELETE`
- 路径：`/booking/{id}`
- 功能：删除指定 booking
- 鉴权：是
- 资源前置条件：需要已存在 booking
- 结果语义：删除成功后再次读取应返回资源不存在
```

要求：

- 接口清单描述“接口能做什么”。
- 不要求在接口清单里写完整 `Body`、`Headers`、`Expected` 脚本。
- 若请求体字段较复杂，可用合法 JSON 样例补充，但这属于接口说明，不等于测试步骤。

## 资源来源与依赖链规则

资源生命周期是本规范中的核心约束。

### 通用规则

- `detail / update / patch / delete` 只有在资源来源明确时才可独立成完整能力链路。
- 资源来源只能是以下三类之一：
  - 同链路 `create`
  - 同资源 `list / detail` 获取并保存 id
  - 文档显式声明预置资源

### 推荐写法

#### 写法 1：同链路 create

```md
**资源来源:** 通过 `POST /booking` 创建，并以返回的 `bookingid` 作为后续接口目标
```

#### 写法 2：同资源 list/detail 获取 id

```md
**资源来源:** 通过 `GET /booking` 获取列表中的 `bookingid`
```

#### 写法 3：预置资源

```md
**资源来源:** 使用预置资源 `booking_id=1`
```

### 不推荐的写法

```md
### Scenario: 修改 booking

涉及接口：
- `PATCH /booking/{id}`
```

问题：

- 没说明 `id` 从哪里来
- 没说明是否需要鉴权
- 没说明该场景是独立能力还是依赖前序链路

## 预期效果写法

预期效果是文档中最重要的“可验证功能语义”，系统应基于它自动生成断言。

推荐写法：

```md
## 预期效果

- 健康检查应返回服务存活状态
- 创建成功后应返回新资源标识
- 详情读取应返回与创建或更新后一致的关键字段
- 全量更新后所有目标字段应按新值覆盖
- 部分更新后未修改字段应保持不变
- 删除后再次读取应返回资源不存在
- 列表过滤应返回合法数组结果
```

不要求写成：

- 每一步 HTTP 状态码断言
- 每个字段逐项断言
- 手工 `save_context` / `uses_context`

这些应优先交给工具完成。

---

## 可选增强信息

以下信息不是规范主体，只在需要提高生成稳定性时补充：

- `**Request:**`
- `**Body:**`
- `**Headers:**`
- `**Expected:**`
- `**save_context:**`
- `**uses_context:**`

适用场景：

- 接口字段命名不规则
- 资源链路较复杂，工具容易推断错误
- 需要对某一高价值链路做更稳定的生成控制

原则：

- 常规文档优先写功能和关系
- 只有在工具推断不稳时，再补这些增强信息

## 好示例与坏示例

### 对照 1：功能描述导向

好示例：

```md
### Scenario: Booking 生命周期管理

覆盖 booking 的创建、详情读取、更新与删除能力。

**涉及接口:** `POST /booking`、`GET /booking/{id}`、`PUT /booking/{id}`、`PATCH /booking/{id}`、`DELETE /booking/{id}`

**关键依赖:** 更新、删除、详情读取都依赖已存在 booking

**资源来源:** 默认通过 `POST /booking` 创建 booking
```

坏示例：

```md
### Scenario: Booking 测试

测试 booking 相关接口。
```

问题：

- 没说明覆盖哪些接口
- 没说明资源依赖
- 没说明这是哪类能力链路

### 对照 2：预期效果导向

好示例：

```md
## 预期效果

- 创建成功后应返回 booking 标识
- 删除后再次读取应返回资源不存在
```

坏示例：

```md
## 预期效果

- 接口应正常
```

问题：

- 不可验证
- 无法稳定转成断言
